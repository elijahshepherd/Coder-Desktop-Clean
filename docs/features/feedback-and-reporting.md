Feedback and automatic bug reporting were added in Coder Desktop `0.0.15` and remain part of the current app. They help useful product signals reach the public GitHub issue tracker without asking users to write a full report by hand.

> [!IMPORTANT]
> Reports are public GitHub issues. They should never include provider keys, local credentials, private file contents, raw chat history, or personal identity.

## Message Feedback

Assistant messages show feedback actions on hover:

- Copy.
- Like.
- Dislike.
- Response timing.

Copy writes the full assistant message to the clipboard.

Like and dislike open a small optional note field. When the user submits feedback, Coder Desktop creates a GitHub issue with:

- The feedback type.
- The app version.
- The operating system.
- The active provider.
- A trimmed user message.
- A trimmed assistant message.
- The optional feedback note.
- A sanitized response duration.

The issue uses the `Like` or `Dislike` label when GitHub accepts the label.

## Automatic Bug Reports

The automatic reporter watches for real application failures, including:

- Main-process uncaught exceptions.
- Main-process unhandled promise rejections.
- Renderer errors.
- Renderer unhandled promise rejections.
- Renderer process crashes.
- Renderer load failures.
- Provider failures that are not expected setup states.
- Provider status codes returned by chat providers.
- Provider no-output cases after retries.
- Update check failures.
- Internet tool exceptions.
- Workspace tool exceptions.
- Image generation failures.
- Hourly diagnostic scan findings.

Expected setup states such as a missing API key or a disabled provider are not treated as bugs.

Automatic reports are background reports. When direct GitHub issue creation is not available, Coder Desktop saves a pending sanitized report locally instead of opening an unexpected browser window.

## Manual Bug Reports

Settings includes a `Report bug` button. The user can add an optional note and submit a manual report from inside the app.

Manual bug reports include:

- The optional note.
- The app version.
- The operating system.
- The active provider state.
- Sanitized system information.
- Recent sanitized report-log events.

If direct GitHub issue creation is not available, manual reports can open a prefilled GitHub issue link so the user can finish the public report.

## Hourly Diagnostics

Coder Desktop runs a lightweight diagnostic scan roughly once per hour while the app is open.

The scan records healthy checks and reports real findings such as:

- A large backlog of pending issue reports.
- Stored active-chat state that no longer points to an existing chat.
- A failed diagnostic scan run.

The scan does not read private project files, provider keys, or raw chat history.

## Privacy Filtering

Before a report is sent, Coder Desktop filters the body.

Filtering includes:

- API key redaction.
- Authorization token redaction.
- Password and secret field redaction.
- User home path redaction.
- Stack trimming.
- Chat text trimming.
- Recent report-log trimming.
- Fixed report labels and titles.

> [!CAUTION]
> Sanitization reduces risk, but public issue reporting should stay focused on technical diagnostics. Do not add raw private files or secrets to feedback notes.

## Spam Protection

Reports are protected by:

- A report fingerprint.
- A twenty-four-hour duplicate cooldown per fingerprint.
- A daily report limit.
- Fixed labels generated by the app.
- Sanitized titles that cannot be controlled directly by the user.

This keeps reporting useful without allowing repeated prompts or repeated failures to flood the issue tracker.

## GitHub Authentication

Coder Desktop opens issues from the user's computer. It does not use a server.

Issue creation uses:

- `CODER_DESKTOP_GITHUB_TOKEN`, `GITHUB_TOKEN`, or `GH_TOKEN` when available.
- Stored Git credentials for `github.com` when environment tokens are not available.
- A locally saved pending report and prefilled GitHub issue link when direct issue creation is not available.

If none of those paths can be prepared, the report status is saved as failed in the app state instead of exposing private data elsewhere.
