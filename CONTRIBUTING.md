# Contributing

Coder Desktop is source-available for public inspection and read-only for public contribution by default. The repository is meant to be understandable, auditable, and useful to learn from, but public pull requests are not accepted unless Elijah Shepherd explicitly invites a specific change.

## What You Can Do

You are welcome to:

- Read the source.
- Inspect the architecture.
- Review release files and manifests.
- Open issues for clear bug reports.
- Report documentation problems.
- Report release download problems.
- Share security concerns through the security reporting process.

Good reports help the project even when code contributions are not open.

## Pull Requests

Please do not open a pull request unless the maintainer has asked for one. Uninvited pull requests may be closed without review.

If a change is invited, it should:

- Stay focused on the requested issue.
- Match the existing architecture.
- Avoid unrelated refactoring.
- Include verification steps.
- Avoid empty or temporary release files.
- Preserve the desktop app design direction.
- Preserve the UI capitalization standard.

## Issue Reports

Useful issue reports include:

- What happened.
- What you expected to happen.
- Steps to reproduce.
- Operating system.
- App version or commit when known.
- Screenshots or logs when helpful.
- Whether the issue happens in a packaged desktop build or development mode.

Avoid posting API keys, private project files, secrets, or sensitive logs.

## Documentation Feedback

Documentation issues are welcome. A good documentation report explains:

- Which document is unclear.
- What information is missing.
- What the reader was trying to do.
- What would have made the answer easier to find.

## Security Reports

Security issues should follow [SECURITY.md](SECURITY.md). Do not publish exploit details in a public issue before the maintainer has had a reasonable chance to review the report.

## Project Direction

Coder Desktop is maintained with a specific product direction:

- Local-first desktop application.
- Calm Codex-like workspace.
- Explicit security controls.
- Real release downloads.
- Clear documentation.
- Read-only public contribution posture.

Feature suggestions are welcome as discussion, but they do not imply acceptance or implementation.
